## Task 12: Port `renderChartConfig.ts` to emit Trino SQL

**Goal:** `packages/common-utils/src/core/renderChartConfig.ts` emits Trino SQL using `TrinoSchemaSerializer` from Phase 1.1 Task 5. No ClickHouse functions in emitted output. Search and Dashboards become functional against Athena.

**Files (modify):**
- `packages/common-utils/src/core/renderChartConfig.ts` (~1500 lines, the main rewrite)
- `packages/common-utils/src/__tests__/renderChartConfig.test.ts` (assertions update)
- `packages/common-utils/src/__tests__/__snapshots__/renderChartConfig.test.ts.snap` (~50 snapshots)
- `packages/common-utils/src/core/metadata.ts` (if it generates SQL fragments; review)
- `packages/common-utils/src/core/searchChartConfig.ts` (uses chSql for WHERE composition; review)
- `packages/common-utils/src/core/histogram.ts` (time-bucket SQL)
- `packages/common-utils/src/core/materializedViews.ts` — **delete** (no MV support in Berg)

**Acceptance criteria:**
- [ ] `renderChartConfig` no longer imports anything from `../clickhouse/`
- [ ] No emitted SQL contains: `toStartOfInterval`, `fromUnixTimestamp64Milli`, `parseDateTimeBestEffort`, `argMaxMerge`, `quantile(...)(...)`, `quantilesMerge`, `JSON_VALUE` (the ClickHouse form, not Trino's `json_extract_scalar`), `toUnixTimestamp64Milli`
- [ ] Time bucketing uses `date_trunc('unit', col)` for fixed-unit buckets; sub-unit (e.g., 5-min) buckets use `date_trunc('minute', col) - INTERVAL (date_diff('minute', '1970-01-01', col) % 5) MINUTE` or equivalent rounding pattern
- [ ] Aggregations: `count`, `sum`, `avg`, `min`, `max`, `count(distinct x)`, `approx_percentile(x, p)`. The HyperDX-specific `argMax(value, ts)` for "latest value" survives (Trino-native).
- [ ] Materialized-view paths in `renderChartConfig` are deleted; chart config has no `mvColumn`/`materializedViews` consumers.
- [ ] All existing `renderChartConfig.test.ts` cases pass with Trino-shaped expected SQL. Snapshots regenerated and committed.
- [ ] Manual smoke: open `/search?source=<id>` → type empty Lucene query → results table renders. Open `/chart` → create a chart on a saved Source → renders. Open a dashboard tile → renders.
- [ ] `make ci-lint` passes.
- [ ] `make ci-unit` passes (538 + ~1335 tests, 0 failures).

**Verify:** `cd packages/common-utils && yarn ci:unit src/__tests__/renderChartConfig.test.ts` and manual browser smoke (boot dev stack, run Search, run a chart).

**Steps:**

- [ ] **Step 1: Inventory the ClickHouse function usage.**

```bash
cd packages/common-utils
grep -rn "chSql\|toStartOfInterval\|fromUnixTimestamp64Milli\|parseDateTimeBestEffort\|argMaxMerge\|quantile(" src/core | head -30
```

Build a function-by-function mapping table (CH → Trino) before any code change.

- [ ] **Step 2: Pick the chart-config emit strategy.**

Two viable approaches:

**Approach A — `TrinoSchemaSerializer` for everything.** Pass the full chart config to a new method `TrinoSchemaSerializer.renderChart(config, schema)` that emits the full SELECT. Highest reuse with the Lucene parser; biggest refactor.

**Approach B — Trino-flavored template literals.** Replace `chSql` calls with plain string templates that compose Trino SQL. Lower abstraction, more direct port. Use `TrinoSchemaSerializer.serialize()` for just the WHERE clause derived from Lucene.

Pick B for v1: minimal architectural change, easier to review, easier to debug. The SerializerV2 → Trino architectural cleanup belongs in a later refactor.

- [ ] **Step 3: Add a `chartUtils.ts` Trino helper module.**

Create `packages/common-utils/src/core/chartUtils.ts` exposing:

```ts
export function quoteIdent(name: string): string { /* ... */ }
export function fullyQualifiedTable(catalog: string, database: string, table: string): string { /* ... uses toAthenaCatalogName */ }
export function timeBucket(column: string, granularity: string): string { /* date_trunc / floor-by-N-units */ }
export function timeRangeWhere(column: string, startMs: number, endMs: number): string {
  return `${quoteIdent(column)} BETWEEN from_unixtime(${Math.floor(startMs/1000)}) AND from_unixtime(${Math.floor(endMs/1000)})`;
}
export function aggregateExpr(fn: string, args: string[]): string {
  // Map ClickHouse aggregate names to Trino equivalents.
}
```

The `quoteIdent` and `fullyQualifiedTable` here replace the duplicates in `packages/app/src/components/Catalog/sql.ts` — move the source of truth into common-utils so chart code and catalog code agree.

- [ ] **Step 4: Port `renderChartConfig.ts` body, function by function.**

Walk the file top to bottom. For each `chSql\`...\`` template:
- Identify the SQL fragment it builds
- Replace with a plain template literal using helpers from Step 3
- Adjust function arguments / time math for Trino
- Drop any branch that handles `metricTables` or `materializedViews`

Same for `searchChartConfig.ts` and `histogram.ts`.

- [ ] **Step 5: Replace the snapshot tests.**

```bash
cd packages/common-utils
yarn ci:unit -u src/__tests__/renderChartConfig.test.ts
```

This regenerates all snapshots. Visually inspect the snapshot diff to confirm the new SQL is sensible Trino. Commit the new snapshots.

- [ ] **Step 6: Update non-snapshot assertions.**

`renderChartConfig.test.ts` has explicit `expect(sql).toContain('toStartOfInterval(...)')` style assertions. Replace each with the Trino-shaped expected substring.

- [ ] **Step 7: Delete `core/materializedViews.ts` if no longer referenced.**

```bash
grep -rn "from '@/core/materializedViews'\|materializedViews" packages/common-utils/src
```

Delete the file and the test.

- [ ] **Step 8: Run lint + unit; iterate until both pass.**

```bash
yarn build:common-utils
make ci-lint
make ci-unit
```

The app package's render-time consumers (`DBSearchPage`, `DBDashboardPage`, etc.) call into `renderChartConfig` for SQL. Most failures will be lint-level (unused imports, type narrowing). Fix as they appear.

- [ ] **Step 9: Manual smoke test in browser.**

```bash
yarn build:common-utils
yarn app:dev
```

Login → Catalog → save a Source on a real S3 Tables table → `/sources/list` → open in Search → Lucene query empty → row table loads. Open `/chart` → make a chart. Open `/dashboards/list` → load any dashboard.

- [ ] **Step 10: Commit.**

```bash
git add -A
git commit -m "feat(common-utils): port renderChartConfig.ts to Trino SQL emit

ClickHouse-flavored chart-config emitter (chSql, toStartOfInterval,
```
