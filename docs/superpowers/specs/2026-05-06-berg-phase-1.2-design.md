# Berg Phase 1.2 — Data-Layer Trino Migration

**Status:** Following Phase 1.1 (Tasks 1-11 + cleanup PRs). Architecture decisions inherited from `docs/superpowers/specs/2026-05-04-berg-design.md`.

**One-line summary:** Finish the ClickHouse → Trino/Athena data-layer migration that Phase 1.1 deferred. Unblocks Search and Dashboards (currently broken at runtime against Athena) and removes the backward-compat scaffolding that's been carrying us.

**Spec:** [`docs/superpowers/specs/2026-05-04-berg-design.md`](./2026-05-04-berg-design.md) is the canonical Berg architecture. This spec is a Phase 1.2 delta only.

---

## 1. What's broken right now

After Phase 1.1, the following user surfaces work end-to-end against real S3 Tables via Athena:
- Login / register / team management
- Catalog browse (catalogs → databases → tables → schema)
- Catalog → Sample tab (50-row preview)
- Catalog → Stats tab (row count, partition info)
- Catalog → DDL tab (synthetic CREATE TABLE)
- Save Source from Catalog; Sources list / edit / delete

The following surfaces are broken at runtime:
- **Search / Discover** (`/search?source=...`) — Lucene parses fine and posts to `/v1/query`, but `renderChartConfig.ts` emits ClickHouse SQL (`toStartOfInterval`, `fromUnixTimestamp64Milli`, `argMaxMerge`, `parseDateTimeBestEffort`, etc.) which Athena rejects with `Function ... not registered` or `Invalid catalog specified` errors.
- **Dashboards / Charts** (`/dashboards/...`, `/chart`) — same root cause, same chart-config plumbing.
- **SQL Editor** (`/clickhouse`) — separate path, deferred to Phase 1.3.

## 2. Goals (Phase 1.2)

- Search renders results end-to-end against Athena
- Dashboards / Chart Explorer render charts end-to-end against Athena
- The transitional `clickhouse/` shim and the legacy `SourceKind` aliases come out of `common-utils`
- The `Source.from`-synthesis shim comes out of `packages/app/src/source.ts`
- `HYPERDX_*` env-var prefix renamed to `BERG_*` everywhere
- Stale `CLICKHOUSE_HOST/USER/PASSWORD` and `OPAMP_*` references in CI/E2E env files removed

## 3. Non-goals (deferred to Phase 1.3)

- SQL Editor migration / replacement — separate code path. Phase 1.3 will decide between (A) SQL toggle in Search, (B) minimal `/sql` page, (C) full ClickhousePage port.
- The `QueryExecutionContext` refactor (sending `catalog`/`database` as separate fields) — becomes redundant once `renderChartConfig.ts` emits proper Trino. Deferred unless Task 12 proves it's needed.
- Real Glue federation (`GetCatalogs` API enumeration) — env-var-driven config is fine for v1.
- Phase 2 features (writes, roles, alerts) — design unchanged; still v2 territory.

## 6. Acceptance for "Search works"

After Task 12:
- A user with a saved Source pointing at a real S3 Tables table can:
  - Open `/search?source=<id>`
  - Type a Lucene query (e.g., `level:error` or empty for "show all rows")
  - See a histogram (if `timestampColumn` is set) and a results table
  - Click a row → side panel with full row JSON
  - Save as a Saved Search; reload it
- The network requests to `/v1/query` carry Trino SQL with no ClickHouse-specific functions (`toStartOfInterval`, `argMaxMerge`, etc.)
- Athena `ResultReuseConfiguration` is hit on a re-run within the TTL window (response carries `cached: true` semantics if surfaced)
- Cost line at the bottom shows scanned-bytes from the Athena response

## 7. Risks

- **Test snapshot churn:** `renderChartConfig.test.ts` has ~50 snapshot tests asserting the exact ClickHouse SQL output. Every one needs Trino-shaped expected output. The test churn is mechanical but high-volume; allocate review time.
- **Time-bucketing edge cases:** ClickHouse's `toStartOfInterval(t, INTERVAL N UNIT)` supports arbitrary integer multiples of a unit (every 5 minutes, every 12 hours). Trino's `date_trunc('unit', t)` only accepts fixed units. For chart granularities other than 1-unit buckets, we'll need `date_trunc` plus modular arithmetic, or just round to the nearest standard unit. Worth designing.
- **Aggregate function differences:** `argMax(x, t)` exists in Athena/Trino but `argMaxMerge` (state-merge form) does not. The chart config uses both. Need to pick which subset Berg supports.
- **Materialized view paths:** `renderChartConfig.ts` has branches that read `source.materializedViews` and emit reads from MV tables. Berg has no MV concept; those branches should be deleted, not ported.

## 8. Rollback plan

Each task in this phase is its own commit (or small commit chain). If Task 12 introduces regressions in Catalog (the working surface today), git revert the 12 commit; the remaining Catalog flow is fully self-contained and won't be affected.

The transitional shims being deleted in 13/14/15 are removable in either direction — re-add them via revert if anything depends on them post-deletion that the implementation missed.
